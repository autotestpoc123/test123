# TestProd Audit Log Pipeline — PROD Deployment Runbook

**Applies to:** Sprint 2.1 codebase (scheduler + two-phase Analysis/Validate pipeline with mail ledger, cycle guards, and named mutexes)
**Last updated:** 2026-07-19
**Environment:** PROD (`-env PROD`)

---

## 1. System Overview

A biweekly (every second Thursday, anchored to `ScheduleAnchor`) audit pipeline with two phases:

| Phase | Entry | What it does |
|---|---|---|
| Phase 1 — Analysis | `Invoke-TestProdAuditScheduler.ps1 -Phase Analysis` → `Invoke-AuditLog.ps1` | Runs enabled tasks (`mail` → `TestProd_mail_analysis.ps1`, `device` → `TestProd_devicelog_analysis.ps1`), sends BU report emails (ledger-deduplicated), writes run artifacts. |
| Phase 2 — Validate + Archive | `Invoke-TestProdAuditScheduler.ps1 -Phase Validate` → `Invoke-AuditValidate.ps1` | Validates that all expected files are present in the source folder, copies them to the backup share, optionally deletes source files (guarded cleanup), sends failure notifications. |

Supporting scripts:

- `Invoke-TestProdAuditOpsReminder.ps1` — cron-driven ops reminder (preflight + email); advisory, always exits 0 in scheduled mode.
- `Invoke-BuMailResend.ps1` — the **only approved** operator escape hatch to resend a BU email from `sent-emails.json`.
- `Verify-ContentHashStability.ps1` — static gate ensuring email Subject/Body construction is deterministic (required for ledger dedup).
- `TestProd_analysis_comm.psm1` — shared module (config/root resolution, preflight, ledger, LDAP, mail, cycle math).
- `analysis_task.config.psd1` — the configuration data file.

---

## 2. Deployment Package

Deploy the following into **one flat folder** (all scripts resolve siblings via `$PSScriptRoot`), e.g. `D:\Apps\TestProd_audit\`:

```
Invoke-TestProdAuditScheduler.ps1
Invoke-AuditLog.ps1
Invoke-AuditValidate.ps1
Invoke-TestProdAuditOpsReminder.ps1
Invoke-BuMailResend.ps1
Verify-ContentHashStability.ps1
TestProd_mail_analysis.ps1
TestProd_devicelog_analysis.ps1
TestProd_analysis_comm.psm1
analysis_task.config.psd1
modules\ImportExcel\          <-- bundled ImportExcel module (folder, required)
```

> **⚠ Config filename gotcha:** `Resolve-AuditConfigPath` defaults to `analysis_task.config.psd1` (**dot**, not underscore) next to the scripts. If your repo copy is named `analysis_task_config.psd1`, rename it at deploy time, or pass `-ConfigPath`, or set `TestProd_AUDIT_CONFIG_PATH`. A wrong name = "Config file not found" at startup.

> **⚠ ImportExcel:** `Invoke-AuditLog.ps1` hard-fails if `modules\ImportExcel` is missing whenever any device-type task exists. The current config enables `device-msms-member-records`, so the module **is required** in PROD.

## 3. Prerequisites (PROD server)

### 3.1 Platform
- Windows Server with **Windows PowerShell 5.1+** (`#Requires -Version 5.1`).
- .NET assemblies available: `System.DirectoryServices.Protocols`, `System.Security` (present on standard Windows Server).

### 3.2 Certificates (LocalMachine\My store)
`Get-Cert` searches `Cert:\LocalMachine\My` by `CN=<KeyName>` and picks the latest `NotAfter`. The **service account must have read access to the machine store and the certs' private keys**.

| Certificate (CN match) | Used by | Purpose |
|---|---|---|
| `TestProd-audit-prod-cert` | Scheduler/reminder notifications; mail analysis SMTP | Ops notification + BU mail send auth (config `Notification.PROD.CertName`) |
| `cod_TestProd_ntfy_prod@abc.com.cn` | Device analysis | Vault cert-auth + SMTP send identity |

Verify before go-live:
```powershell
Get-ChildItem Cert:\LocalMachine\My |
  Where-Object { $_.Subject -match 'TestProd-audit-prod-cert|cod_TestProd_ntfy_prod' } |
  Select-Object Subject, NotAfter, HasPrivateKey
```
Confirm `HasPrivateKey = True` and `NotAfter` is well beyond the next cycle.

### 3.3 Network / endpoint reachability

| Endpoint | Port | Used by |
|---|---|---|
| `mailhost.ms.com` | 2587 (SMTP) | Notifications + mail-analysis BU emails |
| `mta-hub.cod.ms.com.cn` | 2587 (SMTP) | Device-analysis BU emails |
| `https://vault.srv.ms.com.cn` | 443 | Device analysis — `Get-VaultSecret` (namespace `msms/core`, Eonid `309843`, env `prod`) |
| `cod.ms.com.cn` | 363 (LDAP) | Device analysis — user lookup (`Negotiate` auth with vault-retrieved credential, domain `COD`) |
| `\\cod.test.com.cn\apptest\TestProd_audit_log_backup` | SMB | `BackupRoot` — Phase 2 archive target |

### 3.4 File system (service account permissions)

| Path (from config) | Access | Purpose |
|---|---|---|
| `C:\addin_deploy_cert\TestProd_audit_log\source` (`SourceFolder`) | Read (+Delete if cleanup enabled) | Staged input files; validated + archived in Phase 2 |
| `C:\addin_deploy_cert` (`InputRoot`) | Read | Root for token `{InputRoot}` task inputs |
| `C:\SysAdmin\log` (`LogRoot`) | Read/Write | Output root; runs land under `<LogRoot>\TestProd_audit_log\runs\`, ledger under `<LogRoot>\TestProd_audit_log\ledger\mail-ledger.jsonl` |
| `BackupRoot` UNC + `<endDate>` subfolder | Read/Write/Create | Archive copies |

### 3.5 Vault
- Service identity `cod_TestProd_ntfy_prod` must be authorized for cert login and read of `msa/data/secret/309843/prod/cod_TestProd_ntfy_prod@abc.com.cn`.

## 4. Configuration Review (PROD go-live checklist)

Open `analysis_task.config.psd1` and confirm each item:

- [ ] **`ScheduleAnchor`** — must be a **Thursday** in `yyyyMMdd` and must be **on or before** go-live date, aligned so the biweekly parity produces the correct 2-week/4-week alternation (`cycleIndex` even → `CurrentRunWeeks='2'`, odd → `'4'`). The scheduler *throws* if the anchor is not a Thursday or is in the future.
- [ ] **`Notification.PROD`** — `From`, `OpsTeam`, `CcRecipients`, `CertName` are the real PROD values (current file contains placeholder addresses such as `TestProd-audit-prod@corp.com` / `ops-team@corp.com` — replace before go-live).
- [ ] **`SourceFolder` / `InputRoot` / `LogRoot` / `BackupRoot`** — real PROD paths; note current `BackupRoot` points at a `\\cod.test.com.cn\apptest\...` share, which looks like a test target — confirm the PROD share.
- [ ] **`SourceCleanup`** — currently `Enabled = $false` (source files are copied to backup but **retained**). If PROD should delete source files after verified backup, set `Enabled = $true` and ensure `AllowedRoots` contains the parent of `SourceFolder` (cleanup refuses to delete anything outside `AllowedRoots`; a mismatch yields `CleanupAborted`). Currently `AllowedRoots` lists a UNC path while `SourceFolder` is a local `C:\` path — these do not match; fix one or the other before enabling cleanup.
- [ ] **`Tasks`** — confirm `Enabled` flags. Currently enabled: `mail-msms` (mail) and `device-msms-member-records` (device). All others disabled. Task **names must be unique** (enforced at startup).
- [ ] **`BackupValidationRules`** — file name templates match what ops actually delivers each cycle (`{startDate}`, `{endDate}`, `{endDatePlus1MMdd}` tokens). `TwoWeekFixedFiles` vs `FourWeekFixedFiles` sets must match the 2-week/4-week cycle deliverables.
- [ ] **`EnforceBackupValidation` / `EnforceFailure`** — validation exit-code behavior on differences. Default validation still exits 1 on mismatch; `-FailOnDifference` / `EnforceFailure` additionally writes an error record.
- [ ] **`ReminderTargetTimes`** — `Analysis = '08:00'`, `Validate = '16:00'`: these are the deadlines shown in reminder emails; keep in sync with the scheduled-task times below.
- [ ] **Recipient tables in analysis scripts** — PROD BU recipient maps are hard-coded in `TestProd_mail_analysis.ps1` (PROD block) and `TestProd_devicelog_analysis.ps1`. Review both PROD branches; the mail script's PROD block currently mirrors QA-style test addresses (`abc.test@dev.mocktest.com.cn`) — confirm real PROD recipients before go-live.

## 5. Deployment Steps

1. **Freeze & stage.** Copy the package (Section 2) to the target folder. Confirm the config file name resolves (see gotcha above).
2. **Unblock files** (if copied from a network location):
   ```powershell
   Get-ChildItem D:\Apps\TestProd_audit -Recurse | Unblock-File
   ```
3. **Static gate — hash stability (must PASS):**
   ```powershell
   .\Verify-ContentHashStability.ps1 -FailOnWarning
   # Expect exit code 0. Exit 1 blocks deployment (ledger dedup would be unreliable).
   ```
4. **Config parse check:**
   ```powershell
   Import-PowerShellDataFile .\analysis_task.config.psd1 | Out-Null
   ```
5. **Cert / SMTP / share checks** (Section 3 commands; plus `Test-Path '\\...\TestProd_audit_log_backup'` and `Test-NetConnection mailhost.ms.com -Port 2587`).
6. **Dry preflight (no side effects beyond a preflight report + possible ops email):**
   ```powershell
   .\Invoke-TestProdAuditScheduler.ps1 -env PROD -Phase Analysis
   ```
   - With no staged input files, expect **exit 3** (preflight failed), a report at `<LogRoot>\TestProd_audit_log\runs\preflight_<ts>\preflight-report.json`, and a preflight email to `OpsTeam`. This simultaneously smoke-tests config resolution, cert lookup, and SMTP.
   - Off-cycle days emit `NotThursday` / `OffCycle` warnings — expected during deployment testing.
7. **Register scheduled tasks** (service account, "Run whether user is logged on or not", highest privileges, start-in = deploy folder):

   | Task | Trigger | Command |
   |---|---|---|
   | Ops reminder — Analysis | Every Thursday, before 08:00 (e.g. 07:00) | `powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\Invoke-TestProdAuditOpsReminder.ps1 -Phase Analysis -Environment PROD` |
   | Scheduler — Analysis | Every Thursday 08:00 | `powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\Invoke-TestProdAuditScheduler.ps1 -env PROD -Phase Analysis` |
   | Ops reminder — Validate | Every Thursday, before 16:00 (e.g. 15:00) | `...\Invoke-TestProdAuditOpsReminder.ps1 -Phase Validate -Environment PROD` |
   | Scheduler — Validate | Every Thursday 16:00 | `...\Invoke-TestProdAuditScheduler.ps1 -env PROD -Phase Validate` |

   Weekly triggers are safe: on off-cycle Thursdays the reminder self-skips (`today != cycle endDate` → exit 0), and the scheduler's cycle guards prevent duplicate work. The `NotThursday`/`OffCycle` warnings are informational.
8. **First-cycle supervised run.** On the first scheduled Thursday, have ops stage the required source files, then monitor both phases live (Section 7).

## 6. Exit Codes & Guards (for monitoring)

### `Invoke-TestProdAuditScheduler.ps1`
| Code | Meaning | Action |
|---|---|---|
| 0 | Success | None |
| 1 | Phase failed, or a concurrent run holds the mutex (`Global\TestProdAudit-Analysis` / `Global\TestProdAudit-Validate`) | Check `workflow.log`; do not re-trigger while a run is in flight |
| 2 | Cycle guard: phase already completed for this cycle | Usually a double-click / duplicate trigger; no action |
| 3 | Preflight failed (missing/invalid input files) | Ops stages missing files, re-run the phase; preflight email already sent |

### `Invoke-AuditValidate.ps1` (surfaced through scheduler)
| Code | Meaning |
|---|---|
| 0 | Validation passed; archive `Success` / `NoOp` / `NoSourceFiles` |
| 1 | Validation differences (missing/unexpected files) — scheduler sends **ValidationFailed** email |
| 2 | Archive failure (`BackupFailed` / `CleanupAborted` / `CleanupPartiallyFailed`) — scheduler sends **ArchiveFailed** email; retry is allowed without force flags |

### Force flags (engineering only, audit-logged)
`-ForceRerun` (re-run completed Analysis) and `-ForceRerunArchive` (re-archive a completed cycle) are hidden parameters; every use emits a loud `!!! FORCE_RERUN_USED ... !!!` line with user/host. **Not for operators.** Operators use `Invoke-BuMailResend.ps1` for missing BU emails.

## 7. Post-Deployment / Per-Cycle Verification

Artifacts under `<LogRoot>\TestProd_audit_log\`:

- `runs\latest-run.json` — pointer to the latest analysis run (`RunId`, `RunStatus`).
- `runs\<RunId>\workflow.log`, `run-summary.json`, `run-summary.txt` — per-run status; each task under `runs\<RunId>\tasks\<task>\` (`summary.json`, `task.log`, `report.csv`, `sent-emails.json`).
- `runs\<RunId>\validation\backup-validation-summary.json` — Phase 2 result (`ValidationPassed`, `ArchiveStatus`, `MissingFiles`, `UnexpectedFiles`, `Notification` block).
- `runs\latest-preflight.json` + `runs\preflight_<ts>\preflight-report.json` — most recent preflight failure.
- `ledger\mail-ledger.jsonl` — append-only BU send ledger (dedup evidence).
- Backup share: `<BackupRoot>\<endDate>\` contains all expected cycle files (SHA256-verified copies).

Per-cycle sign-off checklist:
- [ ] Scheduler Analysis exit 0; `run-summary.json` shows all enabled tasks `Success`.
- [ ] BU report emails received (check `sent-emails.json` and ledger entries for the cycle).
- [ ] Scheduler Validate exit 0; `ValidationPassed = true`, `ArchiveStatus = Success` (or `NoOp` where applicable).
- [ ] Backup folder `<BackupRoot>\<endDate>` populated.
- [ ] No `notification-failure.json` sidecar in the run's `validation\` folder or `runs\` root.

## 8. Common Failure Playbook

| Symptom | Likely cause | Fix |
|---|---|---|
| `Config file not found` at startup | Filename dot/underscore mismatch or wrong `TestProd_AUDIT_CONFIG_PATH` | Rename to `analysis_task.config.psd1` or fix the env var |
| `SourceFolder is not configured` | Config key missing and env var unset | Set `SourceFolder` in config |
| Exit 3 + preflight email | Ops hasn't staged cycle files | Stage files matching the templated names exactly (tokens resolve with the cycle's `startDate`/`endDate`), re-run phase |
| `Notification config not available or cert not found` | `TestProd-audit-prod-cert` missing from LocalMachine\My or no private-key access | Install/permission the cert; runs still proceed but emails are skipped |
| `Failed to get vault token` (device task) | Cert-auth failure to vault / network | Check `cod_TestProd_ntfy_prod@abc.com.cn` cert, vault ACL, connectivity |
| Task rejected with "content-diff" | Same cycle re-run produced different email content | Investigate source data; per-BU resend via `Invoke-BuMailResend.ps1`; full re-run requires engineering `-ForceRerun` |
| `CleanupAborted` | Source file outside `SourceCleanup.AllowedRoots` | Align `AllowedRoots` with the actual `SourceFolder`; safe retry allowed |
| Exit 1 "mutex held" | Overlapping trigger while previous run in progress | Wait for completion; check for a hung PowerShell process |

## 9. Rollback

1. Disable the four scheduled tasks.
2. Restore the previous script folder from backup (take a copy of the current folder before deploying — step 0 of any change).
3. Do **not** delete or edit `<LogRoot>\TestProd_audit_log\runs\` or `ledger\mail-ledger.jsonl` — the ledger and cycle-guard state must survive rollback so duplicate BU emails are not sent by the rolled-back version.
4. Re-enable scheduled tasks after restore; the cycle guards make re-triggering the current cycle safe (exit 2 if already complete).

---

## 10. Change Control Notes

- Any change to email Subject/Body construction **must** re-pass `Verify-ContentHashStability.ps1` — the ledger dedup depends on `SHA256(Subject + Body)` determinism.
- Changing `ScheduleAnchor` mid-flight shifts cycle parity (2-week vs 4-week) — coordinate with ops and update `BackupValidationRules` expectations in the same change.
- Task `Name` values are referenced by `BackupValidationRules.DynamicFiles.SummaryTaskName` and by the resend tool — renaming a task breaks both; treat renames as breaking changes.
